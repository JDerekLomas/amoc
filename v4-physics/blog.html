Building an Ocean Circulation Simulator

A development log of building a real-time global ocean model in the browser with WebGPU.
Derek Lomas & Claude · April 27, 2026

The Goal

Build an interactive simulation of Earth's ocean circulation — the system of currents, gyres, and overturning cells that redistributes heat across the planet and drives our climate. The simulation should run in real-time in a web browser using WebGPU, be initialized from real observational data, and allow interactive experimentation (What happens if you melt the ice? Change CO₂? Close the Panama Canal?).

The Physics

The model solves the barotropic vorticity equation on a 1024×512 equirectangular grid:

∂ζ/∂t + J(ψ,ζ) + β·∂ψ/∂x = curl(τ) - r·ζ + A·∇²ζ + buoyancy + drag

Where ζ is vorticity, ψ is the streamfunction (related by ∇²ψ = ζ), β is the Coriolis gradient, τ is wind stress, and the buoyancy term couples density (from temperature and salinity) to the flow.

Temperature, salinity, and sea ice evolve as coupled fields. A 1-layer atmosphere (planned) will bridge land and ocean heat exchange.

Development Iterations

Iteration 1: Grid Upgrade (360×180 → 1024×512) DONE

April 27, 2026 — 8:00 PM

Upgraded from the old 360×180 grid to 1024×512 (power of 2 for future FFT). Switched data loading to the data/ directory with high-resolution datasets. Fixed north-first data orientation (all data files store row 0 = 79.5°N, but the simulation grid has j=0 = south).

Key issue: The iterative SOR Poisson solver doesn't converge at 1024×512 — needs ~750 iterations (spectral radius 0.994). Solution: coarse-grid Poisson at 512×256 with bilinear upsampling.

Iteration 2: Parameter Audit DONE

April 27 — 9:30 PM

Audited all simulation parameters against physical values. Key fix: the thermal expansion / haline contraction ratio (α/β) was 0.063 — should be 0.27 from real seawater. Salinity was 4× too dominant vs temperature, suppressing the thermal-driven AMOC. Also fixed ice insulation (was 10-30× too weak) and ice growth rates.

Iteration 3: Atmosphere Model (Failed) REVERTED

April 27 — 10:30 PM

Added a 1-layer atmosphere with Jacobi smoothing to transport ocean heat inland. The atmosphere worked on a coarse 128×64 grid for performance.

Critical failure: The ATM_FEEDBACK_SST=0.003 parameter caused the atmosphere to gradually freeze all oceans. The coarse Jacobi smoothing averaged air temp to a global mean of ~10-15°C, then fed that back into SST, dragging tropical SST from 28°C toward 10°C. This took ~20 commits to diagnose and required full revert to the last known working state.

Lesson: Physics changes must be tested individually. The atmosphere, deep layer removal, omega change, and Poisson iteration changes all interacted to produce cascading failures.

Iteration 4: GODAS Initialization DONE

April 27 — 11:30 PM

Instead of starting from rest (ψ=0, ζ=0), initialize circulation from GODAS observed surface currents. Compute ζ = ∂v/∂x - ∂u/∂y from observed (u,v), then use the CPU FFT Poisson solver for the exact initial ψ. This eliminates 125,000 steps of spinup time.

Adjust S_solar, A_olr, B_olr so that the equilibrium SST matches observations after accounting for circulation cooling. Target: equator ~27°C, 30° ~22°C, 60° ~6°C.

Iteration 7: Observed Wind on GPU PLANNED

Add observed ERA5 wind stress curl as a GPU storage buffer. Currently the GPU shader uses analytical cos(3φ) which misses realistic gyre patterns. This is the highest-impact physics improvement.

Iteration 8: GPU FFT Poisson Solver PLANNED

Replace coarse-grid SOR (512×256, 40 iterations = 80 dispatches) with exact FFT solver (12 dispatches). Would give sharper western boundary currents and 6× fewer GPU dispatches for the Poisson step. Shaders exist in the simamoc branch but need wiring. (Issue #43)

Architecture

Single-file simulation engine (v4-physics/sim-engine.js, ~4400 lines) running on WebGPU with CPU fallback. GPU compute shaders handle the vorticity timestep, temperature/salinity evolution, boundary conditions, and Poisson solve. CPU handles particles, ice, land rendering, and diagnostics during readback.

The Poisson solve (∇²ψ = ζ) is the computational bottleneck. At 1024×512, iterative SOR can't converge, so we downsample to 512×256, solve there, and bilinearly upsample back. This is a trade-off: western boundary currents get smoothed, but the solver converges in 40 iterations.
